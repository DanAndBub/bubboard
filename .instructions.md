# AI Agent Instructions for Bubboard

This file contains specific guidance for AI agents (Claude, OpenClaw, etc.) contributing to this repository.

## Core Principles

1. **Type Safety First**: Always use TypeScript strict mode. No `any` types.
2. **Explicit Over Implicit**: Prefer clear, verbose code over clever shortcuts.
3. **Atomic Commits**: Each commit should be one logical change.
4. **Security by Default**: Assume all file content might contain secrets.
5. **Read Before Write**: Understand existing patterns before creating new code.

## Before You Start Any Task

1. **Understand the context**
   - Read CONTRIBUTING.md for conventions
   - Read SECURITY.md for security requirements
   - Look at similar code in the project
   - Check recent commits to understand current patterns

2. **Ask clarifying questions**
   - If a requirement is ambiguous, ask in the PR or issue
   - Don't assume implementation details
   - Propose your approach before coding

3. **Check if something already exists**
   - Search for similar functionality
   - Don't duplicate code or logic
   - Extend existing patterns rather than create new ones

## Code Creation Workflow

### Step 1: Plan
```
Before writing any code:
1. Identify the change type (feat/fix/refactor/etc.)
2. Determine scope (scanner/analyzer/ui/etc.)
3. List files that need changes
4. Sketch the types/interfaces needed
5. Plan test cases
```

### Step 2: Create with Types
```typescript
// ALWAYS start with types/interfaces
interface MyFeatureInput {
  agentId: string;
  threshold: number;
}

interface MyFeatureOutput {
  result: string;
  confidence: number;
}

// THEN implement with strong typing
export function myFeature(input: MyFeatureInput): MyFeatureOutput {
  // implementation
}
```

### Step 3: Test
```typescript
// Write tests alongside implementation
describe('myFeature', () => {
  it('should handle normal input', () => {
    const result = myFeature({ agentId: 'test', threshold: 0.5 });
    expect(result.confidence).toBeGreaterThan(0);
  });

  it('should handle edge cases', () => {
    const result = myFeature({ agentId: '', threshold: 0 });
    expect(result).toBeDefined();
  });
});
```

### Step 4: Follow Conventions
```bash
# Use Conventional Commits
git commit -m "feat(scope): description

Body explaining why and what."

# Format: feat|fix|refactor|chore|security|test|docs(scope): description
```

### Step 5: Verify
```bash
# Before pushing:
npm test                 # All tests pass?
npm run lint             # No linting errors?
npm run build            # Builds successfully?
npx tsc --noEmit         # No type errors?
```

## Project Structure Guide

### Adding a Feature

```
New feature → Create in appropriate scope:

Scanner improvements     → src/scanner/
Parser logic            → src/lib/parser.ts
Scoring/health logic    → src/lib/scoring.ts
Cost analysis          → src/lib/cost-tracking/
Core types             → src/lib/types.ts
UI components          → src/components/
API routes             → src/app/api/
Tests                  → __tests__/ adjacent to code
```

### When to Refactor

Refactor when:
- Code is duplicated (DRY principle)
- Function is >30 lines (too complex)
- Type safety could be improved
- Performance is measurably better

Don't refactor when:
- It's someone else's code without discussion
- The task doesn't require it (scope creep)
- Tests might break (unless fixing tests)

## Handling Sensitive Data

### Never Do This
```typescript
// ❌ NEVER hardcode credentials
const apiKey = "sk-1234567890abcdef";

// ❌ NEVER console.log file contents
console.log(fileContent);

// ❌ NEVER return raw secrets in responses
return { config: fullConfig };
```

### Always Do This
```typescript
// ✅ Use environment variables
const apiKey = process.env.AIRTABLE_API_KEY;

// ✅ Redact before displaying
const safe = redactSensitiveValues(jsonContent);

// ✅ Return only needed data
return { config: { name: config.name } };
```

### Red Flags (Stop and Ask)
- File contains `api_key`, `apikey`, `token`, `secret`, `password`
- User config includes credentials
- JSON has `*.json` extensions with sensitive patterns
- Building a feature that stores/transmits user data

When you see these: **Do not proceed.** Ask in the issue/PR what the safe approach is.

## Git Workflow for AI Agents

### Branching
```bash
# Start with master
git checkout master
git pull origin master

# Create feature branch
git checkout -b feat/your-feature-name

# NEVER force push (use --force-with-lease if necessary)
git push --force-with-lease

# Better: keep commits clean to avoid force push
git rebase origin/master
git push origin feat/your-feature-name
```

### Commits
```bash
# ✅ Good: Atomic, semantic commits
git commit -m "feat(analyzer): add config validation"
git commit -m "test(analyzer): add tests for validation"
git commit -m "docs(analyzer): update README"

# ❌ Bad: Lumped together
git commit -m "stuff"
git commit -m "more changes"
```

### Handling Conflicts
```bash
# When conflicts happen:
1. Read ALL conflict markers (don't skip any)
2. Understand both sides of the conflict
3. Make a logical decision
4. Test the resolution
5. Commit with explanation

# Example commit message
git commit -m "fix: resolve merge conflict in analyzer.ts

Chose new scoring algorithm (feature branch) as it's
more maintainable, added fallback to old algorithm
for backwards compatibility."
```

## Testing Expectations

### Unit Tests
- One describe block per function/feature
- Test happy path, edge cases, error cases
- Use descriptive test names
- Aim for >80% coverage on logic

### What to Test
```typescript
// ✅ Test core logic
describe('calculateScore', () => {
  it('returns valid score 0-10', () => {});
  it('handles empty input', () => {});
  it('returns null for invalid input', () => {});
});

// ❌ Don't test implementation details
describe('calculateScore', () => {
  it('calls internal helper function', () => {});
  it('uses specific variable names', () => {});
});
```

### Before Committing
```bash
npm test
# ✅ All tests pass? Commit.
# ❌ Tests fail? Fix them before committing.
```

## Code Review Expectations

When you create a PR:

   ```

   fix: handle missing AGENTS.md gracefully
   docs: expand security troubleshooting section
   ```

2. **PR description** explains:
   - What changed and why
   - Any breaking changes
   - How to test it

3. **Code quality**

   - No TypeScript errors
   - No hardcoded secrets

4. **Documentation**

   - Update README if user-facing

## Common Mistakes to Avoid

### Mistake 1: Using `any`
```typescript
// ❌ DON'T
function process(data: any): any {
  return data;
}

// ✅ DO
function process(data: AgentMap): HealthReport {
  return calculateHealth(data);
}
```

### Mistake 2: Not Handling Errors
```typescript
// ❌ DON'T
const config = JSON.parse(jsonString);

// ✅ DO
try {
  const config = JSON.parse(jsonString);
  return config;
} catch (error) {
  console.error('Invalid JSON:', error);
  return null;
}
```

### Mistake 3: Committing Secrets
```bash
# ❌ DON'T (pre-commit hook will catch this)
echo "API_KEY=sk-12345" > .env
git add .env
git commit -m "add config"

# ✅ DO
cp .env.example .env.local
# Edit .env.local with real values
git add .gitignore        # Contains .env.local
```

### Mistake 4: Large Commits
```bash
# ❌ DON'T
git add .
git commit -m "refactor everything"

# ✅ DO
git add src/lib/analyzer.ts
git commit -m "refactor(analyzer): simplify path matching logic"
git add src/lib/__tests__/analyzer.test.ts
git commit -m "test(analyzer): update tests for new signature"
```

### Mistake 5: Not Testing
```bash
# ❌ DON'T
git push origin feat/my-feature
# (without running tests)

# ✅ DO
npm test           # Pass?
npm run lint       # Pass?
npm run build      # Success?
git push origin feat/my-feature
```

## When You're Stuck

**STOP and ask questions. Don't guess.**

```typescript
// Example: Unclear what type should be
// ❌ DON'T guess
export function calculateHealth(input: any): any {
  return { score: 0 };
}

// ✅ DO ask in PR/issue
"I'm implementing calculateHealth. Should it:
1. Accept AgentMap or just the files array?
2. Return a number or HealthReport object?
3. Handle missing files gracefully or throw?"
```

## Quick Reference: File Types

### Types & Interfaces
- Location: `src/lib/types.ts`
- Establish contracts between modules
- Change sparingly; document breaking changes

### Core Logic
- Location: `src/lib/`
- Pure functions where possible
- Fully tested and typed

### React Components
- Location: `src/components/`
- Server components by default
- `'use client'` only when needed
- Extract logic to `src/lib/`

### API Routes
- Location: `src/app/api/`
- Input validation
- Error handling
- Rate limiting (future)

### Tests
- Adjacent to code: `__tests__/`
- Use Vitest syntax
- Name: `[name].test.ts`
